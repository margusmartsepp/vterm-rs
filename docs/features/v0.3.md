# v0.3 — playbooks and batching

Introduced **playbook orchestration**: a single AI request triggers a complex sequence
of actions across multiple independent terminal sessions.

## Skills

### `batch` — the playbook

Execute a list of `SkillCommand` objects in sequence. A single batch can target multiple
terminal IDs, so dependencies are respected (DB up before backend, backend up before
frontend).

### `wait` — the synchronizer

Pause batch execution for N milliseconds. Bridge for processes that need time to bind
ports or initialise state.

## Workflows unlocked

- **Project boot**: Redis + Postgres + dev server, three windows, one command.
- **Build chain**: `cargo build` → `docker build` → `deploy`, each gated on the previous.

## Compatibility

All v0.2 skills (`screen_read`, `screen_write`, `screen_control`) work inside batches.

> **Historical note (v0.6 retro):** this version emitted N+1 wire lines per batch (N
> sub-responses + 1 summary), which caused off-by-one bugs in client-side response
> counting. v0.5 promised a fix; v0.6 delivered it.
