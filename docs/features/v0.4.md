# v0.4 — performance and synchronisation

Turned the orchestrator into a high-precision automation engine. Every command is timed
internally; synchronisation is content-aware rather than sleep-based.

## Internal performance tracking

Every command's `CommandResult` carries `duration_ms`. No external stopwatch needed.
Useful for measuring build times, API latency, environment startup overhead.

## `WaitUntil` — jQuery-style synchronisation

Replaces `Wait { ms }` with a condition-aware pause. Polls the screen for a substring
pattern with a safety timeout.

```json
{"type": "WaitUntil", "payload": {"id": 1, "pattern": "Compiled successfully", "timeout_ms": 30000}}
```

## Robust batch error handling

`stop_on_error` boolean on `Batch`. When `true`, the batch aborts on the first failure.

> **Historical note (v0.6 retro):** even with `stop_on_error = false`, the batch should
> still report aggregate failure status — v0.4 left it reporting `success`. v0.6 fixes this.

## Visibility control

`Batch` accepts a top-level `visible` flag inherited by every `Spawn` inside it (unless
the `Spawn` overrides). Flip one flag to switch a microservice fleet between "show me
everything" debug mode and headless CI mode.

## Standardised protocol

Every command returns a `CommandResult`:

| Field         | Type    | Notes                              |
| ------------- | ------- | ---------------------------------- |
| `status`      | string  | `"success"` or `"error"`           |
| `duration_ms` | number  | Wall time of this command          |
| `id`          | number? | Terminal id, when applicable       |
| `content`     | string? | Screen data (for `ScreenRead`)     |
| `error`       | string? | Detailed error if `status=error`   |
