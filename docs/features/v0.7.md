# v0.7 — MCP bridge

The v0.7 release transitions `vterm-rs` from a raw JSON-RPC over named-pipe service into a first-class Model Context Protocol (MCP) provider. By building an MCP bridge, we unlock native compatibility with AI agents like Claude Desktop, Cursor, and Cowork.

## v0.7.15 features

### `vterm-rs-python-mcp` (PyO3 SDK)

The standout feature of v0.7.15 is the high-performance Python bridge. Built with PyO3, this SDK provides direct access to the orchestrator with near-zero overhead. 

*   **Operator Methods**: Users can build complex batches using a clean, typed syntax (e.g., `client.spawn_op()`, `client.write_op()`).
*   **Plug-and-Play Server**: The package now includes a built-in MCP server (`vterm-mcp`) that can be used directly by AI clients like Claude Desktop or Cursor.

### v0.7.15: The "Agentic Upgrade"

Based on direct feedback from AI agents (Claude 3.5 Sonnet) using the orchestrator, we implemented several critical P0/P1 features to eliminate "blind spots" in autonomous workflows.

- [x] **Restore `wait_until` (P0)**: Restored the missing pattern-matching tool to the MCP bridge.
- [x] **Full Scrollback (P0)**: `read(history=True)` now iterates the `vt100` buffer to return history beyond the visible screen.
- [x] **Process Monitoring (P1)**: `get_process_state(id)` returns `running` status and `exit_code`.
- [x] **Dimension Control (P2)**: `spawn` now accepts `cols` and `rows` to prevent word-wrap issues.
- [x] **Environment Injection (P2)**: `spawn` supports an `env` dictionary for cleaner session setup.
- [x] **Self-Aware Metadata**: `get_info()` provides version and build timestamps for the agent.

### Atomic Batch API

To eliminate "Are we there yet?" polling, we've introduced an atomic `Batch` command. This allows an agent to send a sequence of actions (e.g., Spawn -> Write -> WaitUntil -> Read) in a single round-trip.

*   **Atomic execution**: Commands in a batch are processed sequentially with microsecond latency between them.
*   **Stop-on-error**: Optional logic to halt the batch if a critical command (like `Spawn`) fails.

### One-Click Installation

The release includes a dedicated **"Install in Cursor"** deep-link, utilizing `uvx` to provide a seamless, one-click experience for IDE integration.

## Headline changes

### `vterm-mcp` Binary

We've introduced a new binary target to the workspace: `vterm-mcp`. This binary leverages the `rmcp` 1.5.0 crate to implement a fully compliant stdio-based MCP server. 

Instead of exposing the PTY directly via MCP logic, `vterm-mcp` acts as a multiplexing client proxy. It converts incoming MCP tool calls into `Request` structures and passes them to the existing `vterm.exe` orchestrator over the `\\.\pipe\vterm-rs-skill` named pipe.

### Live Screen Streaming

A core requirement for effective AI terminal usage is seeing the screen state. When an AI uses the `terminal_wait_until` tool (which blocks until a regex matches on screen or a timeout occurs), the bridge now utilizes `rmcp`'s `NotificationContext` (`Peer<RoleServer>`) to stream live terminal updates back to the AI client over MCP progress notifications. 

This enables complex and efficient AI behavior, where the client receives continuous feedback as commands execute in the shell.

### Zero-Config Lifecycles

The bridge provides auto-spawning capabilities. If an AI client launches `vterm-mcp` and the orchestrator is not currently running, the bridge will silently spawn `cargo run -- --headless` in the background and connect to it.

Crucially, because terminal lifetimes in `vterm-rs` are bound to their orchestrator connection, when the AI client exits (e.g. closing Claude Desktop), `vterm-mcp` drops the pipe connection. The orchestrator immediately reaps the associated terminals, ensuring zero resource leaks without requiring explicit cleanup.

## Tool Definitions

The following tools are now exposed via MCP:
- `terminal_spawn`: Bootstraps a PTY and returns the `id`.
- `terminal_write`: Transmits keys to the PTY.
- `terminal_read`: Instantaneous screen buffer read.
- `terminal_wait_until`: Blocking read that streams live output.
- `terminal_close`: Cleans up a specific PTY.
